## Knowledge

今天学到了什么？

- **Windows 与 Linux 环境变量差异**：在 Windows 宿主机下使用 VS Code Dev Containers 挂载用户家目录（如 `.ssh`）时，应当使用 `${localEnv:USERPROFILE}`，而不是 Linux 常用的 `${localEnv:HOME}`。
- **Dev Container 架构最佳实践**：开发多容器（微服务）项目时，不需要在 Compose 中把所有业务微服务都后台拉起。应当只把基础设施（Redis、DB、SFTP）留在 Compose 中默默运行，而把业务代码服务交由开发者在主开发容器终端中手动控制（方便使用 `launch.json` 进行断点调试）。
- **Corepack 的自动化配置**：Node 22 原生自带 Corepack，但为了避免下载包管理器时的交互提示，可以通过在 Dockerfile 中执行 `corepack prepare pnpm@11.10.0 --activate` 提前固化版本，并配合环境变量 `COREPACK_ENABLE_DOWNLOAD_PROMPT=0` 来彻底关闭弹窗。

## Insight

今天最大的认知升级是什么？

- **“部署环境”不等于“开发环境”**：不能盲目地把生产或部署用的 `docker-compose.yml` 原封不动地搬到 Dev Container 中。开发环境的核心是**“可调式性”**和**“掌控感”**。把不需要修改的底层基础设施容器化，把需要频繁修改的业务代码留给主容器终端去自由编译和 debug，才是最高效的微服务本地开发姿势。

## Framework

今天形成了哪些可复用的方法？

- **微服务项目转 Dev Container 架构四步法**：
  1. **提炼主环境**：将核心开发语言（如 Node + Go）整合进一个统一的 `Dockerfile` 作为主开发镜像。
  2. **精简基础设施**：剥离原 Compose 里的业务容器，只保留数据库、缓存等中间件。
  3. **粘合配置**：通过 `devcontainer.json` 的 `"dockerComposeFile"` 和 `"service"` 绑定主环境与中间件。
  4. **环境固化**：将所有全局工具（如 `npm install -g corepack`、`pnpm install`）统统写入构建期的预处理，杜绝任何交互式确认 `[Y/n]`。
