## Knowledge

今天学到了什么？

* **Branch 与 Tag 在 Submodule 中的底层差异**：`.gitmodules` 中的 `branch = xxx` 配置具有强烈的语义限制，**只能用于动态追踪分支**。如果强行写入静态 Tag，Git 在执行 `--remote` 时会报找不到引用的致命错误（`fatal: Unable to find refs/remotes...`）。
* **游离头指针（Detached HEAD）的正常性**：子模块在切换到固定 Tag 或特定的 Commit ID 时，处于游离状态是完全符合预期的。这正是子模块作为“静态第三方依赖库”的标准工作模式。
* **添加子模块的自动化底层**：`git submodule add` 会并行处理三个动作：克隆源码、自动生成 `.gitmodules` 配置、将子模块的 commit 指针注册到主仓库暂存区。

## Insight

今天最大的认知升级是什么？

* **“显式配置”不如“物理指针”稳妥**：对于需要固定的版本（如 `v1.0.0`），试图在 `.gitmodules` 里通过文本声明去约束它，不如直接在子模块里 `checkout` 该 Tag，并**把该 commit 状态直接提交给主仓库（固化指针）**。主仓库对子模块的本质管理方式就是“记录一个特定时刻的哈希值”，顺应这种底层逻辑往往能避免大部分工具链的魔改报错。

## Framework

今天形成了哪些可复用的方法？

### 1. 静态标签（Tag）依赖的子模块同步闭环工作流

当项目的所有子模块都需要对齐到某个固定版本（Tag）时，不要使用 `--remote`，而是采用“主仓拉取 $\rightarrow$ 批量强制切标签 $\rightarrow$ 主仓锁定指针”的组合拳：

```bash
# 1. 克隆主仓库及基础结构
git clone --recursive <主仓库地址>

# 2. 批量让所有子模块获取最新Tag并强制切换（核心复用命令）
git submodule foreach --recursive 'git fetch --tags && git checkout v1.0.0'

# 3. 提交主仓状态，为全员锁定该版本
git add . && git commit -m "chore: pin all submodules to tag v1.0.0" && git push

```

### 2. 标准的子模块添加与版本锁定法

```bash
# Step 1: 命令行声明式添加（拒绝手动改配置）
git submodule add <仓库URL> <本地路径>

# Step 2: 物理切入锁定目标版本（分支或Tag）
cd <本地路径> && git fetch --all --tags && git checkout <目标版本>

# Step 3: 回归主仓双重提交（.gitmodules 配置 + 子模块指针）
cd -
git add .gitmodules <本地路径>
git commit -m "feat: add submodule pinned to <目标版本>"

```
