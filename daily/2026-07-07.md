# Knowledge

今天建立了几个比较重要的知识点：

### 1. 模型不是知识库

模型本质上是一个复杂的数学函数：

> 输入数字 → 输出下一个 Token 的概率。

知识不是存储在数据库里，而是编码在数十亿甚至数千亿参数中。

---

### 2. AI 学习的是统计规律（Pattern）

模型并不知道什么是"猫"、"爱情"或"哲学"。

它学习的是：

> 什么东西更容易一起出现。

因此，大语言模型首先是一个概率模型，而不是传统意义上的"思考机器"。

---

### 3. Transformer 的核心思想

Transformer 并不是按顺序阅读文本。

它允许：

每个 Token 同时关注所有其他 Token。

这就是 Self-Attention，也是现代 LLM 能够处理长文本的重要基础。

---

### 4. 模型进步 ≠ 算法进步

现代模型能力来自多个因素共同作用：

```text
算法（Algorithm）

        ×

数据（Data）

        ×

算力（Compute）

        ×

工程（Engineering）

        ×

训练方法（Training Recipe）
```

任何一个因素发生突破，都可能推动模型能力提升。

---

### 5. 开源与闭源的区别

公开的是：

- Transformer 原理
- 神经网络
- Attention
- 优化算法

真正不公开的是：

- 数据集
- 数据清洗流程
- 训练配方
- 工程优化
- 超参数细节

## Knowledge
今天学到了什么？

1. **nanoGPT 数据流**：原始文本 → 字符级词表（65个字符）→ 数值化 Token → 二进制 `.bin` 文件供模型训练。
2. **Windows 兼容性陷阱**：
   - PyTorch 的 `torch.compile` 在 Windows 下的兼容性较差，小规模实验直接关闭即可。
   - 设备变量（`device`）必须显式定义，不能依赖默认值，否则会遇到 `model.to(device)` 未定义错误。
3. **规模定律（Scaling Laws）核心结论**：
   - 验证集 Loss 与模型参数量、数据量、计算量呈**幂律关系**（双对数坐标下的线性关系）。
   - **Chinchilla 定律**：计算最优时，参数量 ≈ 20 × 训练 Token 数，模型大小和数据量必须等比缩放。

## Insight
今天最大的认知升级是什么？

**从“调参侠”到“实验设计师”的思维转变。**

以前跑模型只关心 Loss 能不能降下来，而 `scaling_laws.ipynb` 揭示了一个更深层的逻辑：**我们可以通过小规模实验，精确预测大规模训练的效果**。这意味着，即使在只有 CPU 的机器上跑 10M 的小模型，这套方法论依然能指导我未来在大规模 GPU 集群上如何分配算力——是堆层数，还是加数据。这种“可预测性”消除了大模型训练的神秘感，让它变成了一门可以计算的工程科学。

## Framework
今天形成了哪些可复用的方法？

1. **nanoGPT 踩坑万能三板斧（针对 Windows/CPU）**：
   - 配置文件中显式添加 `device = 'cpu'`
   - 配置文件中显式添加 `compile = False`
   - 遇到罕见报错时，优先检查配置文件是否覆盖了默认的 CUDA 相关逻辑。

   - 利用拟合公式外推，为下一阶段的实验（如增大 10 倍参数量）提供理论预期。

## Daily Notes

**今天讨论了什么？**
今天主要探讨了如何在国内本地环境中，通过购买海外服务器（以 LisaHost 为例）实现 Claude Code 的稳定调用。讨论从基础的单人代理部署路径，延伸到了本地网络通信原理（为何需要本地客户端），最后深入探讨了团队多人共享情况下的 IP 防封策略与账号额度分配方案。

## Knowledge

**今天学到了什么？**

1. **终端流量转发机制**：购买了海外服务器后，本地终端并不会自动走代理，需要经过 `本地终端 (export) -> 本地代理客户端 (指定端口) -> 海外服务器 -> Claude API` 这一完整的链路。
2. **防封与 IP 质量**：使用双 ISP 工业级 IP 或美国原生住宅 IP（如 LisaHost）可以极大降低 Anthropic 的风控封号风险。
3. **Claude Code 消耗特性**：Claude Code 在运行时会打包大量代码上下文与目录结构，对 Token 的消耗速度远超普通的网页对话。
4. **多用户协同限制**：官方的 `claude` CLI 限制多设备交叉登录，多人并发使用同一节点极易触发“凭据共享”风控。

## Insight

**今天最大的认知升级是什么？**
**“代理节点”不等于“全局网络自由”。**
很多时候我们以为拥有一台海外服务器就能解决一切连接问题，但实际上，应用层（终端 CLI 工具）与网络层（本地到海外服务器的隧道）是需要手动建立桥梁（环境变量）的。此外，在多人使用 AI 开发者工具时，最大的挑战已经从“网络连通性”变成了“账号风控管理”与“Token 成本控制”。

## Framework

**今天形成了哪些可复用的方法？**
**团队海外 AI 工具安全合规调用模型：**

* **低耦合模式（适合分布式团队）**：1 台优质原生 IP 服务器 $\rightarrow$ Xray 多 UUID 分发 $\rightarrow$ 团队成员各自本地连接 $\rightarrow$ 绑定独立 API Key。实现网络通道统一，账号风险隔离。
* **高内聚模式（适合集中式开发）**：高配置海外服务器 $\rightarrow$ 创建多 Linux 子账户 $\rightarrow$ 成员通过 VS Code Remote SSH 接入 $\rightarrow$ 服务端本地执行 Claude Code。实现流量绝对封闭，风控降至最低。
