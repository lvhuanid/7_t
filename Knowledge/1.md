这是一个非常值得深入的方向。

过去大家讨论的是 **Prompt Engineering（提示词工程）**，而现在 AI Agent（如 Claude Code、Codex、Cursor、OpenAI Agents 等）真正比拼的是 **Context Engineering（上下文工程）**。

甚至可以说：

> Prompt 决定 AI 一次回答得怎么样；Context 决定 AI 整个项目做得怎么样。

---

# 一个完整的知识体系

建议按照下面的顺序学习。

```
AI Agent
│
├── Prompt Engineering（如何提问）
│
├── Context Engineering（如何提供信息）
│
├── Spec Engineering（如何定义目标）
│
├── Knowledge Engineering（如何组织知识）
│
├── Memory Engineering（如何长期记忆）
│
└── Workflow Engineering（如何组织工作流）
```

其中，Context Engineering 是最核心的一层。

---

# 一、什么是 Context Engineering？

一句话：

> 把 AI 在当前任务需要知道的一切信息，以最佳方式组织给 AI。

注意：

不是把资料全部丢进去。

而是：

```
正确的信息
+
正确的顺序
+
正确的粒度
+
正确的时机
```

例如：

一个 AI Programmer 写代码。

真正需要的 Context 包括：

```
项目结构

API

Coding Style

业务规则

当前任务

历史修改

已有Bug

测试

Spec

相关文件

依赖关系
```

而不是：

```
整个项目100万token
```

---

真正厉害的人做的是：

> 让 AI 每次都只看到最需要的 Context。

---

# 二、Context 有哪些类型？

一般分成五层。

```
Context

├── User Context
│
├── Project Context
│
├── Task Context
│
├── Runtime Context
│
└── Historical Context
```

分别解释。

---

## ① User Context（用户上下文）

例如：

```
我是创业者

喜欢中文回复

代码写TS

喜欢函数式

不用emoji

喜欢简洁
```

这是长期不变的。

很多 AI 会保存这里。

例如：

```
Memory
```

---

## ② Project Context（项目上下文）

例如：

```
README

目录结构

规范

Architecture

API

Database

Coding Guideline

Design Doc
```

这些属于：

项目知识。

---

## ③ Task Context（任务上下文）

例如：

```
我要开发登录

目标

输入

输出

约束

优先级

Deadline
```

这是一次任务最重要的信息。

---

## ④ Runtime Context（运行上下文）

例如：

AI 当前看到：

```
当前文件

打开的文件

错误

Terminal

Git Diff

Console

Log
```

Cursor 最依赖这个。

---

## ⑤ Historical Context（历史上下文）

例如：

```
昨天修改了什么

为什么这样改

有哪些坑

Bug原因

Review意见
```

这就是很多 Agent 为什么越来越聪明。

---

# 三、如何设计长期 Context？

推荐一种非常经典的分层结构。

```
Memory

├── Identity
│
├── Preference
│
├── Project
│
├── Knowledge
│
├── Experience
│
└── History
```

例如：

```
Identity

我是独立开发者

--------

Preference

喜欢React

--------

Knowledge

已经掌握MCP

--------

Experience

这个Bug以前出现过

--------

History

昨天完成支付模块
```

AI 可以逐层读取。

而不是：

```
100MB聊天记录
```

---

# 四、如何组织知识库？

这是现在最热门的话题。

推荐采用四层结构。

```
Knowledge Base

├── Docs
│
├── Specs
│
├── Patterns
│
└── Examples
```

例如：

```
knowledge/

    docs/

    specs/

    architecture/

    api/

    prompts/

    examples/

    patterns/

    decisions/

    glossary/
```

为什么？

因为 AI 找知识时通常遵循：

```
先找规则

↓

再找例子

↓

最后自己生成
```

而不是：

```
先生成

再看看有没有错
```

# 五、MCP 如何提供 Context？

很多人误解 MCP。

它不是：

> AI 更聪明。

而是：

> AI 能获取更多、更准确的上下文。

例如：

```
AI

↓

MCP

↓

Github

↓

Issue

↓

PR

↓

Database

↓

Slack

↓

Figma

↓

Notion

↓

Browser

↓

Local File
```

MCP 的职责是：

> 在需要的时候，把外部信息变成 AI 的 Context。

例如：

用户说：

```
修Bug
```

AI 实际需要：

```
Git

Issue

PR

Log

代码

数据库

文档
```

这些都是 MCP 拉回来。

所以：

```
Prompt
+

Memory

+

MCP

+

Knowledge

=

Context
```

# 六、Context 与 Spec 的关系

很多人混淆了两者。

实际上：

```
Spec

回答：

要做什么？
```

```
Context

回答：

做的时候需要知道什么？
```

举例：

Spec：

```
开发一个Todo App

支持：

登录

CRUD

搜索

标签
```

Context：

```
React18

Tailwind

已有Layout

API地址

数据库Schema

UI规范

代码规范

已有组件
```

Spec 定义目标。

Context 提供完成目标所需的信息。

可以用一个公式理解：

```
Output = f(
    Spec,
    Context,
    Model
)
```

其中：

* **Spec** 决定方向（What）
* **Context** 提供素材和约束（With What）
* **Model** 决定推理与生成能力（How）

三者缺一不可。

# 七、一个高质量 AI 项目的上下文架构

推荐采用如下目录：

```text
project/
│
├── README.md                # 项目入口
│
├── docs/
│   ├── architecture.md      # 架构说明
│   ├── glossary.md          # 术语
│   ├── api.md               # API 文档
│   └── decisions/           # 架构决策（ADR）
│
├── specs/
│   ├── product-spec.md      # 产品需求
│   ├── feature-a.md
│   └── feature-b.md
│
├── patterns/
│   ├── coding-style.md      # 编码规范
│   ├── ui-patterns.md       # UI 模式
│   └── examples/            # 参考实现
│
├── memory/
│   ├── project-memory.md    # 长期项目记忆
│   ├── bug-log.md           # 常见问题与解决方案
│   └── lessons.md           # 经验沉淀
│
└── prompts/
    ├── planner.md           # 规划提示
    ├── coder.md             # 编码提示
    └── reviewer.md          # 审查提示
```

这样的结构让 AI 能快速定位到规则、需求、示例和历史经验，而不是依赖漫长的聊天记录。

# 八、建议的学习路线

建议按下面的顺序逐步建立体系，而不是只学提示词：

1. **Prompt Engineering**：学习如何清晰表达需求，让 AI 理解任务。
2. **Spec Engineering**：学习如何把模糊想法转化为可执行的需求说明。
3. **Context Engineering**：学习如何组织项目、知识、状态和历史，让 AI 在复杂任务中持续保持高质量表现。
4. **Knowledge Engineering**：构建适合 AI 检索和复用的知识库，而不是单纯堆积文档。
5. **Memory Engineering**：设计长期记忆机制，沉淀经验、决策和偏好。
6. **Workflow Engineering**：把规划、开发、测试、评审等步骤组织成稳定可复用的 AI 工作流。

如果你的目标是 **AI 编程、独立开发或构建 AI Agent**，我建议把 **Context Engineering** 作为未来重点投入的能力。随着模型能力不断提升，真正拉开效果差距的往往不再是单次 Prompt，而是能否持续、准确地为模型提供恰当的上下文。



### 一个完整 Spec 包含什么？

通常包含八部分。
```
Spec

├── Goal
├── Scope
├── Functional Requirements
├── Non-functional Requirements
├── Constraints
├── Acceptance Criteria
├── Edge Cases
└── Out of Scope
```



### Spec 如何分层？
```
大型项目通常不是一个 Spec。

而是树状结构。

Project Spec
│
├── Product Spec
│
├── Architecture Spec
│
├── API Spec
│
├── UI Spec
│
├── Database Spec
│
└── Feature Spec

例如：

specs/

product.md

architecture.md

api.md

database.md

login.md

payment.md

dashboard.md
```
每个 Feature 一个 Spec。
